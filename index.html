VIGIL TM — AML Transaction Monitoring System

Rule-based transaction monitoring system for NBFC and BFSI compliance teams. Transactions are evaluated synchronously against AML rules inside the FastAPI request cycle. Alerts above the risk threshold are created immediately and enriched asynchronously via Celery for PEP, sanctions, and adverse media checks.

  • Designed a rule evaluation engine covering 10 AML scenarios — structuring, high-risk country transfers, PEP transactions, velocity checks, and jurisdiction anomalies — each with configurable risk weights and score thresholds
  • Chose synchronous rule scoring inside the request cycle to ensure immediate alert creation, with async Celery tasks handling enrichment checks (PEP, sanctions, adverse media) after alert insertion
  • Built an append-only audit log enforced at the application layer — every status change, review action, and role decision is recorded as an insert with no update or delete paths exposed
  • Implemented analyst workflows covering RFI creation, investigation notes, and SAR/CTR PDF generation via ReportLab — designed around how compliance teams actually work a case
  • Added Gemini API integration for SAR narrative drafts — used as a writing assist for analysts, not as a decision engine
  • Enforced RBAC across four roles (Analyst, Senior Analyst, Compliance Manager, Admin) with route-level access control and full Docker deployment

KYC · Customer Screening View on GitHub

VIGIL SCREEN — Risk-Based Customer Screening Platform

Customer screening system for AML compliance workflows. Scores each customer against seven weighted parameters and routes cases by risk band — LOW cases are auto-closed, MEDIUM queued for analysts, HIGH and CRITICAL escalated to compliance officers. Single-customer screening runs synchronously; batch CSV screening runs via Celery.

  • Designed a 7-parameter weighted scoring model (name 25pts, ID 20pts, PEP 15pts, DOB 15pts, nationality 10pts, adverse media 10pts, occupation 5pts) — weights chosen to reflect RBI KYC priority ordering, not arbitrarily
  • Used RapidFuzz for name matching with token_sort, token_set, partial, and ratio strategies combined — threshold set at 85% to balance false positive rate against missed matches on transliterated names
  • Separated single-customer screening (synchronous, inline) from batch CSV screening (async Celery pipeline) — a deliberate trade-off to keep single-customer response fast while handling bulk onboarding without blocking
  • Built an explainability layer that returns reason codes, evidence per parameter, and an auditor summary with every screening result — so analysts can understand and defend the score, not just act on it
  • Implemented the same append-only audit pattern as TM — every review, escalation, and close action is a new insert, enforced at application layer
